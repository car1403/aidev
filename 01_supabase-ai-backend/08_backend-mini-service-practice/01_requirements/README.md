# 01. Requirements

이 폴더에서는 백엔드 미니 서비스의 요구사항을 정리합니다.

요구사항은 코드를 작성하기 전에 “무엇을 만들 것인지”를 정하는 문서입니다. 기능을 먼저 명확히 정리하면 FastAPI 엔드포인트, Supabase 테이블, 서비스 로그, 오류 응답 구조를 일관되게 설계할 수 있습니다.

## 이번 단계의 목표

- 미니 서비스가 해결할 문제를 한 문장으로 설명합니다.
- 필요한 API 기능을 목록으로 정리합니다.
- Supabase에 저장할 데이터와 Upstash Redis로 처리할 데이터를 구분합니다.
- mock 구현과 실제 연동 구현의 범위를 나눕니다.
- 다음 단계인 `02_api-design`에서 사용할 요구사항 기준을 만듭니다.

## 서비스 주제

```text
AI 질문 응답 백엔드 미니 서비스
```

사용자가 질문을 보내면 FastAPI 백엔드가 질문을 검증하고, AI 답변을 생성한 뒤, 질문/답변 기록과 서비스 로그를 저장하는 서비스입니다.

처음에는 비용과 설정 부담을 줄이기 위해 mock LLM 함수로 답변을 생성합니다. 이후 실제 LLM 연동이 필요할 때는 01~03 과정의 기본 방향에 맞춰 Gemini API를 우선 사용하고, OpenAI 예제는 선택적으로 비교할 수 있습니다.

## 전체 처리 흐름

```text
사용자 질문 요청
-> FastAPI 요청 검증
-> 사용자 프로필 확인
-> mock LLM 또는 실제 LLM 호출 위치에서 답변 생성
-> 질문과 답변 저장
-> 서비스 로그 저장
-> JSON 응답 반환
```

## 기능 요구사항

| 기능 | 설명 | 우선순위 |
| --- | --- | --- |
| 상태 확인 | `/health`로 서버 실행 상태를 확인합니다. | 필수 |
| 질문 등록 | 사용자의 질문을 받아 처리합니다. | 필수 |
| 요청 검증 | 사용자 id와 질문 내용이 올바른지 확인합니다. | 필수 |
| AI 답변 생성 | 처음에는 mock 함수로 답변을 생성하고, 이후 LLM API로 확장할 수 있게 설계합니다. | 필수 |
| 질문/답변 저장 | 질문과 답변을 Supabase에 저장합니다. | 필수 |
| 기록 조회 | 저장된 질문/답변 목록을 조회합니다. | 필수 |
| 서비스 로그 저장 | API 처리 성공/실패, 오류, 처리 시간을 기록합니다. | 필수 |
| 오류 응답 | 잘못된 요청에 대해 예측 가능한 JSON 오류를 반환합니다. | 필수 |
| 캐시/요청 제한 | 필요 시 Upstash Redis를 활용할 수 있게 설계합니다. | 선택 |

## 데이터 요구사항

| 데이터 | 저장 위치 | 설명 |
| --- | --- | --- |
| 사용자 프로필 | Supabase | 사용자 id, 표시 이름, 선호 언어, 학습 수준 등 |
| 질문/답변 기록 | Supabase | 사용자 질문, AI 응답, 생성 시간 |
| 서비스 로그 | Supabase | API 성공/실패, 오류 메시지, 처리 시간, metadata |
| 임시 캐시 | Upstash Redis | 반복 질문 응답 캐시, 짧은 상태 저장 |
| 요청 제한 카운터 | Upstash Redis | 짧은 시간 동안의 요청 횟수 관리 |

## 요청 검증 기준

| 항목 | 기준 |
| --- | --- |
| `user_id` | 비어 있으면 안 됩니다. |
| `question` | 비어 있으면 안 됩니다. |
| 질문 길이 | 너무 긴 입력을 막기 위해 최대 길이를 정합니다. 예: 500자 |
| 민감 정보 | API Key, 비밀번호, 토큰 같은 값은 저장하거나 로그로 남기지 않습니다. |
| 오류 응답 | 어떤 값이 잘못되었는지 JSON으로 명확히 알려줍니다. |
