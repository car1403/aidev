# Model Parameter Comparison Guide

LLM API를 사용할 때는 모델 이름만 고르는 것이 아니라, 응답 방식을 조절하는 파라미터도 함께 이해해야 합니다.

이 문서는 수업에서 자주 언급하는 `temperature`, `top_p`, `max_tokens`, `thinkingLevel` 같은 설정을 초보자 기준으로 정리합니다.

## 모델 선택 기준

| 기준 | 설명 | 확인할 내용 |
| --- | --- | --- |
| 비용 | 입력과 출력 토큰에 따라 비용이 달라질 수 있습니다. | 무료 범위, 유료 전환 조건, 결제 계정 |
| 응답 속도 | 모델마다 응답 시간이 다를 수 있습니다. | 실시간 응답이 필요한지, 긴 분석이 필요한지 |
| 한국어 품질 | 한국어 설명, 요약, 코드 주석 품질이 모델마다 다를 수 있습니다. | 같은 질문을 여러 모델에 넣어 비교 |
| 코드 작성 능력 | Python, FastAPI, SQL 같은 코드 생성 품질을 확인합니다. | 실행 가능 여부, 오류 처리, 보안 |
| 컨텍스트 길이 | 긴 대화나 긴 문서를 얼마나 잘 처리하는지 봅니다. | 이전 대화 유지, 긴 README 분석 |

## 주요 파라미터

| 파라미터 | 의미 | 초보자 권장 사용 |
| --- | --- | --- |
| `temperature` | 응답의 무작위성과 창의성을 조절합니다. | 정확한 코드 설명은 낮게, 아이디어 생성은 조금 높게 사용합니다. |
| `top_p` | 다음 단어 후보를 얼마나 넓게 볼지 조절합니다. | 처음에는 기본값을 유지하고 `temperature`를 먼저 이해합니다. |
| `max_tokens` | 출력 길이의 최대치를 제한합니다. | 비용과 응답 길이를 통제하기 위해 너무 크게 잡지 않습니다. |
| `thinkingLevel` | 일부 모델 또는 SDK에서 추론 강도를 조절하는 공급자별 설정입니다. | 지원 여부와 이름이 바뀔 수 있으므로 사용 시점의 공식 문서를 확인합니다. |

## 수업용 설정 예시

### 안정적인 코드 설명

```text
temperature: 0.2
top_p: 1.0
max_tokens: 800
```

코드 설명, 오류 분석, API 응답 구조처럼 정확성이 중요한 경우에는 응답이 너무 흔들리지 않도록 낮은 `temperature`를 사용합니다.

### 아이디어 생성

```text
temperature: 0.7
top_p: 1.0
max_tokens: 1200
```

프로젝트 주제, UI 문구, 테스트 케이스 아이디어처럼 다양한 답변이 필요한 경우에는 `temperature`를 조금 높일 수 있습니다.

## Gemini의 thinkingLevel 이해

`thinkingLevel`은 모든 LLM API에 공통으로 있는 표준 파라미터가 아닙니다. 특정 모델 또는 SDK에서 “더 깊게 생각할지, 빠르게 답할지”를 조절하는 방식으로 제공될 수 있습니다.

수업에서는 다음 정도로 이해하면 충분합니다.

```text
낮은 추론 수준
-> 빠른 응답이 필요할 때 유리할 수 있습니다.

높은 추론 수준
-> 복잡한 분석이나 단계적 추론이 필요할 때 유리할 수 있습니다.
```

단, 실제 파라미터 이름, 지원 모델, 비용 영향은 서비스 제공자와 SDK 버전에 따라 달라질 수 있습니다. 실습 시점에는 반드시 공식 문서와 API 콘솔 화면을 확인합니다.

```text
질문:
FastAPI에서 Pydantic 모델을 사용하는 이유를 초보자에게 설명해줘.

비교 1:
temperature를 낮게 설정하고 답변을 확인합니다.

비교 2:
temperature를 높게 설정하고 답변을 확인합니다.

비교 3:

```

- 어떤 설정이 너무 길거나 불안정했나요?
- 실제 API 호출이라면 비용 관점에서 어떤 설정이 안전할까요?
