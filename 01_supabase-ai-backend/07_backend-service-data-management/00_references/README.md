# References

사용자 정보, 대화 이력, 서비스 로그, 피드백 데이터 설계를 위한 보충 자료를 둡니다.

## 핵심 판단 기준

서비스 데이터를 설계할 때는 먼저 아래 질문을 던집니다.

1. 이 데이터는 나중에 다시 조회해야 하나요?
2. 이 데이터는 사용자별로 접근 제어가 필요한가요?
3. 이 데이터는 운영자가 오류 분석에 사용할 수 있나요?
4. 일정 시간이 지나면 사라져도 되나요?

판단 기준:

```text
오래 보관한다 -> Supabase
사용자별 권한이 필요하다 -> Supabase + Auth/RLS
오류 분석에 사용한다 -> Supabase service_logs
짧게 유지한다 -> Upstash Redis
요청 횟수 카운트다 -> Upstash Redis
```

## 서비스 데이터 체크리스트

| 항목 | 확인 질문 |
| --- | --- |
| 사용자 프로필 | 사용자를 식별할 id와 화면 표시 이름이 있는가? |
| 대화 세션 | 여러 메시지를 하나의 대화방으로 묶을 기준이 있는가? |
| 메시지 | user/assistant/system 역할이 구분되는가? |
| 서비스 로그 | 성공/실패, 오류 메시지, 처리 시간이 남는가? |
| 피드백 | 사용자가 응답 품질을 평가할 수 있는가? |
| 임시 상태 | Redis에 둘 데이터와 Supabase에 둘 데이터가 구분되는가? |
