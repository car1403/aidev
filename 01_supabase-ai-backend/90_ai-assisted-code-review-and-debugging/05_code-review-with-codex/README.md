# 05_code-review-with-codex

이 단원에서는 Codex를 활용해 Python, FastAPI, Supabase, Upstash Redis, LLM API 실습 코드를 리뷰하는 방법을 학습합니다.

코드 리뷰는 코드를 바로 고치는 작업이 아닙니다. 먼저 코드가 요구사항을 만족하는지, 실행 중 오류가 날 가능성은 없는지, 데이터가 잘못 저장되거나 민감 정보가 노출될 위험은 없는지 확인하는 과정입니다.

리팩토링이 "코드를 더 읽기 좋게 정리하는 일"이라면, 코드 리뷰는 "무엇이 위험한지 먼저 찾고 우선순위를 정하는 일"입니다.

## 이 단원에서 다루는 내용

- Codex에게 코드 리뷰를 요청하는 방법
- 실행 오류, 요구사항 누락, 보안 위험을 구분하는 방법
- FastAPI endpoint와 Pydantic 모델을 리뷰하는 방법
- Supabase 저장, 조회, 수정, 삭제 코드에서 조건 누락을 찾는 방법
- Upstash Redis 세션, 캐시, rate limit 코드에서 TTL과 key 설계를 확인하는 방법
- Gemini SDK 기본 사용 흐름, REST 보충 예제, OpenAI 선택 사용 흐름에서 비용, 모델명, API key 노출 여부를 점검하는 방법
- 리뷰 결과를 바로 수정하지 않고, 확인 질문과 검증 방법까지 정리하는 방법

## 코드 리뷰의 기본 순서

Codex에게 "이 코드 어때?"라고만 요청하면 답변이 너무 넓어질 수 있습니다. 다음 순서로 리뷰 관점을 지정하면 결과가 훨씬 명확해집니다.

1. 실행 오류 가능성 확인
2. 요구사항 충족 여부 확인
3. API URL, HTTP Method, 요청/응답 구조 확인
4. Supabase, Redis, LLM API 연동 코드 확인
5. API key, token, 사용자 정보 같은 민감 정보 노출 여부 확인
6. 예외 처리와 오류 응답 확인
7. 로그가 필요한 정보만 남기는지 확인
8. 중복 코드, 함수명, 변수명, 주석 확인
9. 테스트 또는 실행 확인 방법 정리

이 순서가 중요한 이유는 단순히 코드가 보기 좋은지보다, 실제로 실행했을 때 데이터가 잘못 저장되거나 서비스가 멈추는 문제가 더 중요하기 때문입니다.

## 리뷰 요청에 포함하면 좋은 정보

코드 리뷰를 요청할 때는 코드만 붙여 넣는 것보다 아래 정보를 함께 제공하는 것이 좋습니다.

```text
다음 코드를 리뷰해주세요.

파일 위치:
C:\aidev\01_supabase-ai-backend\08_backend-mini-service-practice\04_implementation-guide\main_mock.py

이 코드의 목적:
질문을 입력받아 mock-first 응답을 만들고, 서비스 로그를 남기는 FastAPI 예제입니다.

리뷰 우선순위:
1. 실행 오류 가능성
2. endpoint URL과 HTTP Method가 적절한지
3. 요청/응답 모델이 명확한지
4. 오류 응답이 초보자가 이해할 수 있는지
5. 나중에 Supabase 또는 실제 Gemini SDK 호출로 바꾸기 쉬운지

바로 수정하지 말고, 먼저 문제 목록과 이유를 정리해주세요.
```

이렇게 요청하면 Codex는 코드의 목적과 리뷰 우선순위를 알고 답변할 수 있습니다.

## FastAPI 코드 리뷰 기준

FastAPI 코드를 리뷰할 때는 endpoint가 보기 좋게 작성되었는지만 보지 않습니다. 요청이 들어오고 응답이 나가는 전체 흐름이 명확한지 확인해야 합니다.

확인할 항목은 다음과 같습니다.

- endpoint URL이 리소스 중심으로 작성되었는가?
- GET, POST, PUT, DELETE가 의미에 맞게 사용되었는가?
- 요청 Body가 Pydantic 모델로 검증되는가?
- 빈 문자열, 너무 긴 입력, 잘못된 role 값 같은 입력 오류를 막는가?
- 없는 데이터를 조회할 때 404를 반환하는가?
- 서버 설정 문제나 외부 API 오류를 500 계열 오류로 구분하는가?
- Swagger UI에서 예시 입력과 응답 구조를 이해할 수 있는가?

예를 들어 질문 생성 API라면 `/questions` 또는 `/chat/messages`처럼 무엇을 다루는 API인지 알 수 있는 이름이 좋습니다. 반대로 `/do`, `/run`, `/process`처럼 의미가 넓은 이름은 리뷰에서 개선 대상으로 볼 수 있습니다.

## Pydantic 모델 리뷰 기준

Pydantic 모델은 단순히 타입을 적는 곳이 아닙니다. API가 받을 수 있는 데이터의 규칙을 정하는 곳입니다.

확인할 항목은 다음과 같습니다.

- 필수 입력과 선택 입력이 구분되어 있는가?
- `str`, `int`, `list[str]` 같은 타입 힌트가 명확한가?
- 입력 길이 제한이 필요한 필드에 제한이 있는가?
- 예시 데이터가 제공되어 Swagger UI에서 확인하기 쉬운가?
- 응답 모델이 실제 반환 데이터와 일치하는가?
- 중첩 JSON 구조가 너무 복잡하지 않은가?

초보 단계에서는 모델을 너무 복잡하게 만들기보다, 요청 모델과 응답 모델을 분리해서 읽기 쉽게 만드는 것이 중요합니다.

## Supabase 코드 리뷰 기준

Supabase 코드는 데이터가 실제로 저장되는 부분이므로 조건 누락을 특히 주의해야 합니다.

확인할 항목은 다음과 같습니다.

- Supabase URL과 key를 `.env`에서 읽는가?
- service role key가 프론트엔드 코드에 노출되지 않는가?
- insert, select, update, delete 결과가 비어 있을 때 처리하는가?
- update와 delete에 `.eq(...)` 같은 조건이 빠지지 않았는가?
- 테이블명과 컬럼명이 SQL 문서와 일치하는가?
- 사용자별 데이터라면 user_id 또는 owner_id 기준으로 분리되는가?
- 로그 테이블에 민감 정보가 저장되지 않는가?

특히 delete 코드에서 조건이 빠지면 전체 데이터가 삭제될 수 있습니다. 코드 리뷰에서는 이런 위험을 가장 먼저 확인해야 합니다.

## Upstash Redis 코드 리뷰 기준

Redis는 빠르게 읽고 쓰는 임시 저장소입니다. 이 과정에서는 로컬 Redis가 아니라 Upstash Redis를 사용합니다.

확인할 항목은 다음과 같습니다.

- Redis URL과 token이 `.env`에서 읽히는가?
- token이 print, 로그, 프론트엔드 화면에 노출되지 않는가?
- 세션, 캐시, rate limit key가 목적별로 분리되어 있는가?
- 만료가 필요한 데이터에 TTL이 설정되어 있는가?
- 장기 보관이 필요한 데이터는 Redis가 아니라 Supabase에 저장되는가?
- 같은 사용자의 요청인지 구분할 수 있는 key 설계가 되어 있는가?

예를 들어 최근 질문 캐시는 Redis에 잠시 보관할 수 있지만, 최종 대화 이력은 Supabase에 저장하는 것이 좋습니다.

## LLM API 코드 리뷰 기준

01부터 03 과정에서는 Gemini SDK를 기본으로 사용하고, REST 호출은 구조 이해용 보충, OpenAI는 선택 비교 또는 추가 실습으로 유지합니다. 코드 리뷰에서는 어떤 모델을 쓰는지보다, API key와 비용을 안전하게 다루는지가 더 중요합니다.

확인할 항목은 다음과 같습니다.

- 기본 예제가 Gemini SDK 중심으로 설명되어 있는가?
- REST 호출 예제가 기본 경로처럼 섞이지 않고 보충 예제로 분리되어 있는가?
- OpenAI 예제는 선택 실습 또는 비교 실습으로 분리되어 있는가?
- API key가 코드에 직접 적혀 있지 않은가?
- 실제 API 호출 전 key 존재 여부를 확인하는가?
- mock-first 호출과 실제 Gemini SDK 호출이 명확히 분리되어 있는가?
- `provider`, `model`, `actual_api_called`, `llm_call_mode`로 호출 상태를 구분하는가?
- 반복문 안에서 실제 LLM API를 과도하게 호출하지 않는가?
- 출력 토큰, 최대 응답 길이, 재시도 횟수 같은 제한이 있는가?
- API 오류가 발생했을 때 사용자에게 적절한 메시지를 반환하는가?

LLM API 코드는 정상 상황보다 실패 상황을 더 꼼꼼히 봐야 합니다. key가 없거나, 할당량이 부족하거나, 네트워크 오류가 발생해도 서비스가 완전히 멈추지 않아야 합니다.

## 서비스 로그 리뷰 기준

서비스 로그는 나중에 문제가 생겼을 때 원인을 찾기 위해 남기는 기록입니다. 하지만 너무 많은 정보를 남기면 오히려 위험합니다.

확인할 항목은 다음과 같습니다.

- 성공과 실패 로그가 구분되는가?
- `event_type` 같은 로그 유형이 일관되게 작성되는가?
- endpoint, provider, model, actual_api_called, llm_call_mode, item_id처럼 문제 분석에 필요한 정보가 있는가?
- API key, token, 비밀번호, 개인 식별 정보가 로그에 들어가지 않는가?
- 오류 메시지가 너무 기술적이어서 사용자에게 그대로 노출되지 않는가?
- 로그 저장 실패가 전체 API 실패로 이어지지 않도록 처리되어 있는가?

로그는 "나중에 문제를 찾기 위한 정보"이지 "모든 데이터를 저장하는 공간"이 아닙니다.

## Codex 리뷰 요청 예시

아래 예시는 실제 실습 파일을 리뷰할 때 사용할 수 있는 요청 형식입니다.

```text
다음 FastAPI 코드를 리뷰해주세요.

리뷰 대상:
C:\aidev\01_supabase-ai-backend\08_backend-mini-service-practice\04_implementation-guide\main_supabase.py

코드 목적:
사용자 질문을 저장하고, AI 응답과 서비스 로그를 Supabase에 기록하는 백엔드 예제입니다.

리뷰 관점:
1. 실행 오류 가능성
2. FastAPI endpoint 설계
3. Pydantic 요청/응답 모델
4. Supabase insert/select 조건
5. API key와 민감 정보 노출 위험
6. 예외 처리와 오류 응답
7. 초보자가 이해하기 어려운 구조

출력 형식:
- 치명적인 문제
- 수정이 필요한 문제
- 개선하면 좋은 문제
- 확인 질문
- 실행 또는 테스트 방법

아직 코드를 수정하지 말고 리뷰 결과만 정리해주세요.
```

리뷰 결과를 받은 뒤에는 바로 모든 내용을 적용하지 않습니다. 먼저 실제 요구사항과 맞는지 확인하고, 치명적인 문제부터 순서대로 수정합니다.

## 리뷰 결과를 읽는 방법

Codex의 리뷰 결과는 다음 기준으로 나누어 읽는 것이 좋습니다.

| 구분 | 의미 | 처리 방법 |
| --- | --- | --- |
| 치명적인 문제 | 실행 실패, 데이터 손상, key 노출처럼 바로 수정해야 하는 문제 | 가장 먼저 수정 |
| 수정이 필요한 문제 | 요구사항 누락, 잘못된 상태 코드, 조건 누락 등 | 기능 단위로 수정 |
| 개선하면 좋은 문제 | 변수명, 중복 코드, 함수 분리 등 | 시간이 있을 때 정리 |
| 확인 질문 | 코드만 보고 판단하기 어려운 부분 | 요구사항 문서와 비교 |
| 테스트 방법 | 수정 후 확인할 명령이나 시나리오 | 반드시 실행 |

코드 리뷰의 목적은 모든 코드를 완벽하게 만드는 것이 아니라, 중요한 위험을 놓치지 않는 것입니다.

## 실습 흐름

1. 리뷰할 파일을 하나 선택합니다.
2. 파일의 목적을 한 문장으로 정리합니다.
3. 리뷰 우선순위를 정합니다.
4. Codex에게 "바로 수정하지 말고 리뷰만 해달라"고 요청합니다.
5. 리뷰 결과를 치명도별로 분류합니다.
6. 실제 요구사항과 맞는 문제만 선택합니다.
7. 수정이 필요한 경우 다음 단원 또는 별도 요청으로 수정합니다.
8. 수정 후 실행 명령과 테스트 결과를 확인합니다.
