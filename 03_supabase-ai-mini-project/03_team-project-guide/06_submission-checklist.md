## API 설계 문서 확인

- [ ] 엔드포인트 URL이 리소스 중심으로 일관되게 명명되어 있습니다.
- [ ] HTTP Method(GET/POST/PUT/DELETE)가 의미에 맞게 사용되었습니다.
- [ ] 표준화된 에러 응답 형식이 정의되어 있습니다.
- [ ] 4xx/5xx 예외 상황별 처리 규칙이 문서화되어 있습니다.
- [ ] Request/Response용 Pydantic 모델의 필수/선택 필드가 정리되어 있습니다.
- [ ] 타입 힌트와 예시값이 작성되어 있습니다.
- [ ] 중첩 JSON은 nested 모델로 표현되어 있습니다.

## 화면 설계서 확인

- [ ] 메인, 상세, 입력, 설정, 오류 화면의 와이어프레임이 있습니다.
- [ ] 정보 계층 구조(IA)가 논리적으로 정리되어 있습니다.
- [ ] 버튼, 입력폼, 테이블, 차트의 스타일과 배치 기준이 설명되어 있습니다.
- [ ] 클릭, 호버, 드래그, 스크롤, 필터, 새로고침 같은 사용자 액션에 대한 시스템 반응이 정의되어 있습니다.
- [ ] 화면에서 어떤 API를 호출하는지 연결되어 있습니다.

## 데이터베이스 설계서 확인

- [ ] 테이블이 3정규화(3NF) 기준으로 설계되었습니다.
- [ ] 데이터 중복이 최소화되어 있습니다.
- [ ] 이상현상(삽입/갱신/삭제)이 방지되도록 설계되었습니다.
- [ ] 논리 ERD에 엔티티, 속성, 관계, 카디널리티가 표시되어 있습니다.
- [ ] 물리 ERD에 테이블명, 컬럼, PK/FK, 인덱스가 표시되어 있습니다.
- [ ] 모든 컬럼의 데이터 타입, 길이, 제약조건, 기본값, 설명이 작성되어 있습니다.
- [ ] RLS를 사용했다면 정책 의도와 테스트 결과가 문서화되어 있습니다.

## 대시보드 구현 결과물 확인

- [ ] 구현된 대시보드가 실행 환경에서 동작합니다.
- [ ] 로그 수집 기능이 동작합니다.
- [ ] 로그 조회 기능이 동작합니다.
- [ ] 로그 시각화 기능이 동작합니다.
- [ ] 핵심 지표가 화면에서 확인됩니다.
- [ ] 필터 또는 새로고침 기능이 동작합니다.
- [ ] 오류 상황이 사용자에게 안내됩니다.

## Supabase 검증

- [ ] Supabase 프로젝트가 생성되어 있습니다.
- [ ] `.env.example`에 필요한 환경변수 이름이 정리되어 있습니다.
- [ ] 실제 `.env` 파일은 제출물에 포함되지 않았습니다.
- [ ] anon key와 service role key 사용 위치가 구분되어 있습니다.

## 실행 검증

- [ ] FastAPI 백엔드가 실행됩니다.
- [ ] Swagger 문서가 열립니다.
- [ ] Streamlit 프론트엔드가 실행됩니다.
- [ ] 프론트엔드에서 백엔드 API를 호출합니다.
- [ ] 데이터가 Supabase에 저장됩니다.
- [ ] 저장된 데이터를 다시 조회할 수 있습니다.

## AI/UX 검증

- [ ] AI 응답 또는 AI 분석 기능이 있습니다.
- [ ] 프롬프트 설계 의도가 설명되어 있습니다.
- [ ] 사용자 피드백 데이터를 어떻게 반영할지 설명되어 있습니다.
- [ ] 응답 생성 중 loading/spinner/status가 표시됩니다.
- [ ] SSE를 적용했다면 실시간 스트리밍 흐름이 문서화되어 있습니다.

## 발표 전 점검

- [ ] README 실행 순서가 정확합니다.
- [ ] 민감 정보가 제출물에 포함되지 않았습니다.
- [ ] 팀원 역할과 구현 내용을 설명할 수 있습니다.
- [ ] 대시보드의 핵심 기능을 3분 안에 시연할 수 있습니다.
- [ ] 06 과정 이후 Docker/AWS 운영 확장 계획을 설명할 수 있습니다.
