# 05. Workflow Project Rubric

AI Workflow 프로젝트 평가 기준입니다.

이 루브릭은 `09_ai-workflow-mini-project`의 이미지 기준 산출물과 연결됩니다.
화면이나 데모만 제출하는 것이 아니라, 설계 근거와 실행 검증 결과를 함께 제출해야 합니다.

## 평가표

| 항목 | 배점 | 확인 내용 |
| --- | ---: | --- |
| 업무 시나리오 | 10 | 자동화할 기술 지원 업무가 구체적이고 사용자가 명확한가 |
| 노코드 워크플로우 설계서 | 20 | Trigger -> 처리 -> 출력 흐름, 분기/반복/병렬 구조, 외부 서비스 연동, 인증, 오류 처리가 문서화되어 있는가 |
| 데이터 흐름과 Payload | 10 | 노드 간 payload 스키마, Mapper/Formatter, 필수/선택 필드, 예시 데이터가 작성되어 있는가 |
| 도구 선택 | 10 | AIPP, n8n, Dify, FastAPI, Streamlit 중 선택한 도구의 이유가 타당한가 |
| RAG 아키텍처 설계서 | 20 | chunking, overlap, embedding 모델, vector dimension, 거리 계산 방식, metadata, 백업/확장 전략이 설명되는가 |
| 보안과 예외 처리 | 10 | Prompt Injection, 입력/출력 필터링, 인증 실패, API 오류, 검색 결과 없음, 알림 실패 대응이 있는가 |
| 워크플로우 실행 결과서 | 15 | 정상/예외 시나리오 실행 로그, 입력, 기대 결과, 실제 결과, 성공 여부가 정리되어 있는가 |
| 발표와 설명 | 5 | 팀이 설계 근거, RAG 구조, 테스트 결과, 개선 방향을 말로 설명할 수 있는가 |

## 최소 통과 기준

```text
[ ] 기술 지원 자동화 시나리오 1개가 있다.
[ ] Trigger -> 처리 -> 출력 구조가 있다.
[ ] AIPP, n8n, Dify 중 하나 이상을 사용하거나 구체적으로 설계했다.
[ ] RAG 또는 Knowledge 사용 여부와 이유가 설명되어 있다.
[ ] 정상 테스트 1개와 예외 테스트 1개가 있다.
[ ] 최종 산출물 3종이 작성되어 있다.
```

## 피드백 질문

```text
이 워크플로우는 어떤 업무 문제를 해결합니까?
왜 이 도구를 선택했습니까?
어떤 데이터가 어떤 노드로 이동합니까?
RAG를 사용한다면 어떤 문서를 어떤 기준으로 검색합니까?
오류가 발생하면 어떤 fallback이 실행됩니까?
실행 결과를 어떤 로그와 테스트 결과로 증명합니까?
```
