# 01. Topic Selection

이 문서는 팀 프로젝트의 주제를 정하기 위한 활동지입니다.

## 1. 기본 주제

이번 프로젝트의 기본 주제는 아래와 같습니다.

```text
노코드·로코드 기반 기업형 지능형 기술 지원 자동화(Tech Support) 워크플로우 구축
```

팀은 이 큰 주제 안에서 더 구체적인 상황을 선택합니다.

## 2. 선택 가능한 세부 주제 예시

| 세부 주제 | 설명 |
| --- | --- |
| 계정 잠금 해제 지원 | 사용자의 계정 잠금 문의를 접수하고 본인 확인, 해결 절차 안내, 담당자 알림을 처리합니다. |
| 소프트웨어 설치 문의 | 설치 오류 메시지를 분석하고 관련 매뉴얼을 검색해 해결 절차를 안내합니다. |
| 사내 시스템 접속 오류 | VPN, SSO, 권한 문제를 분류하고 담당 부서로 연결합니다. |
| 장애 신고 자동 분류 | 장애 내용을 분석해 긴급도와 담당 팀을 자동 분류합니다. |
| FAQ 기반 기술 지원 | 자주 묻는 질문과 매뉴얼을 RAG로 검색해 답변 초안을 생성합니다. |

## 3. 팀 주제 작성

아래 항목을 팀별로 작성합니다.

| 항목 | 작성 내용 |
| --- | --- |
| 팀명 |  |
| 프로젝트 제목 |  |
| 해결하려는 문제 |  |
| 주요 사용자 |  |
| 사용 도구 | AIPP / n8n / Dify / FastAPI / Streamlit / 기타 |
| RAG 사용 여부 | 사용 / 미사용 / 선택 예정 |
| 외부 서비스 연동 | Slack / Email / Jira / Notion / Database / API / 기타 |

## 4. 좋은 주제의 조건

좋은 주제는 범위가 너무 넓지 않고, 입력과 출력이 분명해야 합니다.

예를 들어 “회사 업무 자동화”는 너무 넓습니다. “사내 VPN 접속 오류 문의를 접수하고, 오류 유형을 분류한 뒤, 해결 매뉴얼을 찾아 답변하고, 필요하면 담당자에게 알리는 워크플로우”처럼 좁히는 것이 좋습니다.

## 5. 강사 확인 질문

- 이 주제는 실제 업무 문제와 연결되어 있습니까?
- 워크플로우가 시작되는 Trigger가 명확합니까?
- 어떤 문서나 데이터가 RAG 검색 대상이 되는지 설명할 수 있습니까?
- 외부 서비스 연동이 필요한 이유가 있습니까?
- 정상 흐름과 예외 흐름을 모두 만들 수 있습니까?
