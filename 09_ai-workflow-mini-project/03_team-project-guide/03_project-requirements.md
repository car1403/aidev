# 03. Project Requirements

이 문서는 `09_ai-workflow-mini-project`의 필수 요구사항입니다.

이번 프로젝트의 목표는 노코드·로코드 도구로 기술 지원 자동화 워크플로우를 설계하고, RAG와 외부 도구 연동, 멀티 에이전트 흐름, 예외 처리, 테스트 결과를 문서로 증명하는 것입니다.

## 1. 필수 구현 또는 설계 범위

| 구분 | 요구사항 |
| --- | --- |
| RAG 및 도구 노드 | 기술 지원 문서, FAQ, 매뉴얼 중 하나 이상을 검색 대상으로 정의하고, 검색 노드 또는 Knowledge 노드의 역할을 설명합니다. |
| 멀티 에이전트 오케스트레이션 | 단일 AI가 모든 일을 처리하지 않고, 분류, 검색, 답변 생성, 검토, 알림 같은 역할을 분리합니다. |
| 운영 안정화 | 인증 실패, API 오류, 검색 결과 없음, 알림 실패 같은 예외 상황에 대한 대응 흐름을 정의합니다. |
| 테스트 및 검증 | 정상 케이스와 예외 케이스를 실행하거나 시뮬레이션하고 결과를 기록합니다. |

## 2. 노코드 워크플로우 설계서 요구사항

작성 파일:

```text
docs/no-code-workflow-design.md
```

반드시 포함할 내용:

- Trigger -> 처리 -> 출력의 전체 흐름
- 사용 도구: AIPP, n8n, Dify, FastAPI, Streamlit 등
- 분기, 반복, 병렬 처리 구조와 사용 이유
- 외부 서비스 연동 노드: Slack, Email, Jira, Notion, Database, API 등
- 인증 방식: OAuth, API Key, Webhook Secret 등
- 오류 처리: 재시도, 대체 경로, 담당자 알림, 실패 로그 기록
- 노드 간 payload 스키마
- Mapper, Formatter, 필드 변환 로직
- 필수/선택 필드와 예시 payload
- 각 연결선(edge)별로 어떤 데이터가 넘어가는지에 대한 설명
- 인증 실패, API 오류, 알림 실패, 데이터 누락 상황별 처리 기준

## 3. RAG 아키텍처 설계서 요구사항

작성 파일:

```text
docs/rag-architecture.md
```

반드시 포함할 내용:

- 사용할 원본 문서의 종류와 특성
- chunking 단위: 문장, 단락, 토큰, 의미 단위 등
- chunk size와 overlap 설정 이유
- embedding 모델 선정 이유
- embedding 모델 후보 비교: OpenAI text-embedding-3 계열, bge 계열, multilingual-e5 계열 등
- vector dimension
- 거리 계산 방식: cosine, euclidean 등
- vector DB 또는 Knowledge 저장소 구조
- 컬렉션/인덱스 명명 규칙
- partition 또는 sharding 필요 여부
- metadata 저장 방식
- 백업/복구 정책
- 대용량 문서 증가 시 확장 전략

## 4. 워크플로우 실행 결과서 요구사항

작성 파일:

```text
docs/workflow-execution-result.md
```

반드시 포함할 내용:

- 주요 정상 시나리오 실행 결과
- 주요 예외 시나리오 실행 결과
- 테스트별 입력 데이터, 기대 결과, 실제 결과, 성공 여부
- 실행 로그 또는 화면 캡처 설명
- API 호출 또는 노드 실행 결과
- RAG 검색 결과가 응답에 반영되는 방식
- 실패 시 재시도, fallback, 담당자 알림 결과
- 테스트 후 발견한 문제점
- 다음 개선 계획

## 5. 평가 기준

| 평가 항목 | 확인 질문 |
| --- | --- |
| 업무 적합성 | 기술 지원 자동화 문제를 구체적으로 해결하고 있습니까? |
| 워크플로우 구조 | Trigger, 처리, 출력, 분기, 반복, 예외 흐름이 논리적으로 연결되어 있습니까? |
| 데이터 흐름 | 노드 간 payload와 필드 변환이 문서화되어 있습니까? |
| RAG 설계 | chunking, embedding, vector DB 구조가 근거와 함께 설명되어 있습니까? |
| 운영 안정성 | 인증, 오류, 알림 실패, 검색 실패에 대한 대응 흐름이 있습니까? |
| 검증 근거 | 실행 로그와 테스트 결과를 제출했습니까? |

## 6. 최소 완성 기준

시간이 부족한 팀은 아래 기준까지는 반드시 완성합니다.

- 기술 지원 자동화 시나리오 1개
- Trigger -> 처리 -> 출력 구조 1개
- RAG 또는 Knowledge 사용 여부 판단
- 외부 서비스 연동 노드 1개 이상
- 정상 테스트 1개
- 예외 테스트 1개
- 최종 산출물 3종 작성
