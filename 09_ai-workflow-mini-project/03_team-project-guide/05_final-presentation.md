# 05. Final Presentation

최종 발표는 구현 화면을 보여주는 시간이기도 하지만, 더 중요한 것은 팀이 설계한 AI 워크플로우를 논리적으로 설명하는 시간입니다.

## 1. 발표 권장 순서

| 순서 | 발표 내용 |
| --- | --- |
| 1 | 팀명과 프로젝트 제목 |
| 2 | 해결하려는 기술 지원 문제 |
| 3 | 전체 워크플로우 흐름 |
| 4 | 사용 도구와 선택 이유 |
| 5 | 노코드 워크플로우 설계 |
| 6 | RAG 아키텍처 설계 |
| 7 | 멀티 에이전트 또는 역할 분리 구조 |
| 8 | 예외 처리와 운영 안정화 전략 |
| 9 | 테스트 결과와 실행 로그 |
| 10 | 개선할 점과 확장 방향 |

## 2. 발표에 포함할 핵심 질문

발표자는 아래 질문에 답할 수 있어야 합니다.

- 사용자의 문의는 어떤 Trigger로 시작됩니까?
- 어떤 노드가 어떤 일을 담당합니까?
- 노드 간 데이터는 어떤 형식으로 전달됩니까?
- RAG를 사용한다면 어떤 문서를 어떤 방식으로 검색합니까?
- 외부 서비스 연동에는 어떤 인증 방식이 필요합니까?
- 실패 상황이 발생하면 어떤 대체 흐름이 실행됩니까?
- 테스트 결과를 통해 무엇을 확인했습니까?

## 3. 발표 자료 구성 예시

```text
1. 프로젝트 개요
2. 문제 상황과 사용자
3. 전체 워크플로우 다이어그램
4. 노드별 역할과 payload
5. RAG 아키텍처
6. 외부 서비스 연동 구조
7. 예외 처리와 운영 안정화
8. 테스트 결과
9. 시연 화면 또는 실행 로그
10. 회고와 개선 계획
```

## 4. 시연 준비

실제 도구 화면을 시연하는 팀은 아래 사항을 미리 확인합니다.

- API Key 또는 OAuth 정보가 노출되지 않도록 처리했는가?
- 테스트용 데이터만 사용하는가?
- 외부 알림이 실제 사용자에게 발송되지 않도록 제한했는가?
- 실패 상황을 보여줄 테스트 입력을 준비했는가?
- 시연이 안 될 경우를 대비해 화면 캡처 또는 실행 로그를 준비했는가?
