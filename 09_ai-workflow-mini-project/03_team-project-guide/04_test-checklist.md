# 04. Test Checklist

이 문서는 팀 프로젝트의 테스트 항목을 정리하기 위한 체크리스트입니다.

테스트는 “잘 실행되었다”라고 말하기 위한 절차가 아닙니다. 어떤 입력을 넣었을 때 어떤 노드가 실행되고, 어떤 결과가 나왔으며, 실패했을 때 어떻게 대응했는지를 확인하는 과정입니다.

## 1. 정상 시나리오 테스트

| 체크 | 항목 |
| --- | --- |
| [ ] | 사용자가 기술 지원 문의를 입력하면 워크플로우가 시작된다. |
| [ ] | 문의 내용이 올바른 카테고리로 분류된다. |
| [ ] | 필요한 경우 RAG 또는 Knowledge 검색이 실행된다. |
| [ ] | 검색 결과가 답변 생성에 반영된다. |
| [ ] | 최종 답변 또는 처리 결과가 사용자에게 전달된다. |
| [ ] | Slack, Email, Jira, Notion 등 외부 서비스 연동이 필요한 경우 정상 호출된다. |
| [ ] | 실행 로그가 남는다. |

## 2. 예외 시나리오 테스트

| 체크 | 예외 상황 | 확인할 내용 |
| --- | --- | --- |
| [ ] | 입력값 누락 | 필수 필드가 없을 때 안내 메시지가 출력되는가? |
| [ ] | 인증 실패 | API Key, OAuth, Webhook Secret 오류를 감지하는가? |
| [ ] | 외부 API 실패 | 재시도 또는 대체 흐름이 실행되는가? |
| [ ] | RAG 검색 결과 없음 | fallback 답변 또는 담당자 연결이 실행되는가? |
| [ ] | 알림 전송 실패 | 실패 로그가 남고 재시도 또는 수동 처리 안내가 있는가? |
| [ ] | 부적절한 입력 | Prompt Injection 또는 민감 정보 입력을 필터링하는가? |

## 3. 노드 간 데이터 흐름 테스트

| 체크 | 항목 |
| --- | --- |
| [ ] | 노드 간 payload 필드명이 문서와 일치한다. |
| [ ] | Mapper 또는 Formatter가 필요한 필드를 올바르게 변환한다. |
| [ ] | 필수 필드와 선택 필드가 구분되어 있다. |
| [ ] | 각 edge별 예시 payload가 작성되어 있다. |
| [ ] | 오류 발생 시 어떤 payload가 다음 단계로 전달되는지 정의되어 있다. |

## 4. RAG 테스트

| 체크 | 항목 |
| --- | --- |
| [ ] | 검색 대상 문서가 명확하다. |
| [ ] | chunk size와 overlap이 문서 특성에 맞다. |
| [ ] | embedding 모델 선정 이유가 있다. |
| [ ] | 검색 결과가 질문과 관련 있는지 확인했다. |
| [ ] | 검색 결과가 없을 때 fallback 흐름이 있다. |

## 5. 실행 결과 기록

테스트 결과는 아래 파일에 정리합니다.

```text
docs/workflow-execution-result.md
```

각 테스트는 최소한 아래 정보를 포함합니다.

- 테스트 이름
- 입력 데이터
- 실행된 주요 노드
- 기대 결과
- 실제 결과
- 성공/실패 여부
- 발견한 문제
- 개선 방향
