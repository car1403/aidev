# 06. Submission Checklist

최종 제출 전에 아래 항목을 확인합니다.

## 1. 필수 제출 파일

팀 프로젝트 폴더에 아래 파일이 있어야 합니다.

| 체크 | 파일 | 설명 |
| --- | --- | --- |
| [ ] | `README.md` | 팀 프로젝트 개요와 실행 방법 |
| [ ] | `docs/project-plan.md` | 주제, 사용자, 문제, 범위 |
| [ ] | `docs/no-code-workflow-design.md` | 노코드 워크플로우 설계서 |
| [ ] | `docs/rag-architecture.md` | RAG 아키텍처 설계서 |
| [ ] | `docs/workflow-execution-result.md` | 워크플로우 실행 결과서 |
| [ ] | `docs/ops-quality-plan.md` | 운영 안정화와 품질 관리 계획 |
| [ ] | `docs/test-checklist.md` | 테스트 체크리스트 |
| [ ] | `presentation/final-presentation.md` | 최종 발표 자료 |

## 2. 노코드 워크플로우 설계서 확인

| 체크 | 항목 |
| --- | --- |
| [ ] | Trigger -> 처리 -> 출력 흐름이 작성되어 있다. |
| [ ] | 병렬, 분기, 반복 구조가 필요한 경우 근거가 작성되어 있다. |
| [ ] | Slack, Email, Jira, Notion, Database, API 연동 노드가 설명되어 있다. |
| [ ] | OAuth, API Key 등 인증 방식이 정리되어 있다. |
| [ ] | 노드별 오류 처리와 재시도 기준이 있다. |
| [ ] | 노드 간 payload 스키마와 예시가 작성되어 있다. |

## 3. RAG 아키텍처 설계서 확인

| 체크 | 항목 |
| --- | --- |
| [ ] | 원본 문서의 종류와 특성이 설명되어 있다. |
| [ ] | chunking 단위, chunk size, overlap 설정 근거가 있다. |
| [ ] | embedding 모델 선정 이유가 있다. |
| [ ] | vector dimension과 거리 계산 방식이 작성되어 있다. |
| [ ] | 컬렉션/인덱스 명명 규칙이 있다. |
| [ ] | metadata, 백업/복구, 확장 전략이 포함되어 있다. |

## 4. 워크플로우 실행 결과서 확인

| 체크 | 항목 |
| --- | --- |
| [ ] | 정상 시나리오 실행 결과가 있다. |
| [ ] | 예외 시나리오 실행 결과가 있다. |
| [ ] | 실행 로그 또는 화면 캡처 설명이 있다. |
| [ ] | 실패 시 재시도, fallback, 담당자 알림 결과가 있다. |
| [ ] | 테스트 후 개선할 점이 작성되어 있다. |

## 5. 제출 전 보안 확인

| 체크 | 항목 |
| --- | --- |
| [ ] | `.env` 파일을 제출하지 않는다. |
| [ ] | API Key, OAuth Client Secret, Webhook Secret을 문서에 직접 적지 않았다. |
| [ ] | 실제 고객 정보나 개인정보를 테스트 데이터로 사용하지 않았다. |
| [ ] | 외부 알림은 테스트 채널 또는 테스트 계정으로만 발송했다. |

## 6. 최종 확인 질문

- 이 프로젝트가 어떤 기술 지원 문제를 해결하는지 1분 안에 설명할 수 있습니까?
- 워크플로우 다이어그램을 보고 각 노드의 역할을 설명할 수 있습니까?
- RAG 구조와 검색 기준을 설명할 수 있습니까?
- 실패 상황에서 서비스가 어떻게 대응하는지 설명할 수 있습니까?
- 실행 결과를 로그와 테스트 결과로 증명할 수 있습니까?
