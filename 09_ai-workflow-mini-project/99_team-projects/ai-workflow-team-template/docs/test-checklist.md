# Test Checklist

이 문서는 팀 프로젝트 제출 전 최종 테스트 체크리스트입니다.

## 1. 노코드 워크플로우 설계서

| 체크 | 항목 |
| --- | --- |
| [ ] | Trigger -> 처리 -> 출력 흐름이 작성되어 있다. |
| [ ] | 분기, 반복, 병렬 처리 구조가 필요한 이유와 함께 작성되어 있다. |
| [ ] | 외부 서비스 연동 노드가 설명되어 있다. |
| [ ] | OAuth, API Key, Webhook Secret 등 인증 방식이 설명되어 있다. |
| [ ] | 각 노드의 입력과 출력이 작성되어 있다. |
| [ ] | 노드 간 payload 스키마와 예시가 있다. |
| [ ] | Mapper 또는 Formatter 변환 규칙이 있다. |
| [ ] | 오류 상황과 fallback 흐름이 작성되어 있다. |

## 2. RAG 아키텍처 설계서

| 체크 | 항목 |
| --- | --- |
| [ ] | 검색 대상 문서가 명확하다. |
| [ ] | chunking 단위, chunk size, overlap이 작성되어 있다. |
| [ ] | chunking 설정 이유가 설명되어 있다. |
| [ ] | embedding 모델 선정 이유가 있다. |
| [ ] | vector dimension과 distance metric이 작성되어 있다. |
| [ ] | 컬렉션/인덱스 명명 규칙이 있다. |
| [ ] | metadata 저장 방식이 정의되어 있다. |
| [ ] | 백업/복구와 확장 전략이 있다. |

## 3. 워크플로우 실행 결과서

| 체크 | 항목 |
| --- | --- |
| [ ] | 정상 시나리오 테스트 결과가 있다. |
| [ ] | 예외 시나리오 테스트 결과가 있다. |
| [ ] | 실행 로그 또는 화면 캡처 설명이 있다. |
| [ ] | RAG 검색 결과가 답변에 반영된 예시가 있다. |
| [ ] | 외부 서비스 연동 결과가 정리되어 있다. |
| [ ] | 실패 시 재시도, fallback, 담당자 알림 결과가 있다. |
| [ ] | 발견한 문제와 개선 방향이 작성되어 있다. |

## 4. 보안 확인

| 체크 | 항목 |
| --- | --- |
| [ ] | `.env` 파일을 제출하지 않는다. |
| [ ] | API Key와 OAuth Secret을 문서에 직접 적지 않았다. |
| [ ] | 테스트 데이터에 실제 개인정보가 없다. |
| [ ] | 외부 알림은 테스트 채널 또는 테스트 계정으로만 발송했다. |

## 5. 발표 준비

| 체크 | 항목 |
| --- | --- |
| [ ] | 프로젝트 문제와 해결 방식을 1분 안에 설명할 수 있다. |
| [ ] | 전체 워크플로우 다이어그램을 설명할 수 있다. |
| [ ] | RAG 아키텍처를 설명할 수 있다. |
| [ ] | 예외 처리와 운영 안정화 전략을 설명할 수 있다. |
| [ ] | 테스트 결과를 근거로 성공 여부를 설명할 수 있다. |
