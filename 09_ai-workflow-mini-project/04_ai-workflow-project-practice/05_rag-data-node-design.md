# 05. RAG Data Node Design

이 문서는 팀 프로젝트에서 RAG, Knowledge, 데이터 처리 노드, 노드 간 데이터 흐름을 설계하기 위한 실습 문서입니다.

RAG를 꼭 구현해야 하는 것은 아닙니다. 하지만 프로젝트에서 지식 검색이 필요한지, 데이터가 어떤 과정을 거쳐 AI 응답으로 이어지는지 설명할 수 있어야 합니다.

## 1. RAG 또는 Knowledge 필요 여부

아래 질문에 답합니다.

```text
우리 프로젝트는 외부 지식이나 문서 검색이 필요한가?
어떤 문서를 참고해야 하는가?
문서가 없다면 어떤 샘플 데이터를 사용할 것인가?
검색 결과가 없을 때는 어떻게 처리할 것인가?
```

작성 예시:

```text
기술지원 FAQ, 장애 대응 매뉴얼, 계정 잠금 처리 절차 문서를 검색 대상으로 사용한다.
검색 결과가 없으면 "담당자 확인 필요" 상태로 분류하고 기본 안내 메시지를 반환한다.
```

## 2. 데이터 처리 노드 설계

입력 데이터가 AI에 전달되기 전에 어떤 처리가 필요한지 정리합니다.

| 단계 | 질문 |
| --- | --- |
| 전처리 | 불필요한 공백, 특수문자, 민감정보를 제거해야 하는가? |
| 분류 | 문의 유형, 긴급도, 담당 부서를 분류해야 하는가? |
| 검색어 변환 | 사용자 질문을 검색에 적합한 키워드로 바꿔야 하는가? |
| RAG 검색 | Knowledge 또는 문서에서 관련 내용을 찾아야 하는가? |
| 후처리 | AI 응답을 JSON, 문장, 알림 메시지 형태로 바꿔야 하는가? |

## 3. 노드 간 데이터 흐름

아래 형식으로 노드 간 데이터 흐름을 작성합니다.

```text
사용자 입력
-> 입력 검증 노드
-> 문의 분류 노드
-> 검색어 변환 노드
-> RAG/Knowledge 검색 노드
-> LLM 답변 생성 노드
-> 출력 검증 노드
-> 사용자 응답 또는 담당자 알림
```

## 4. 입력/출력 계약

각 노드가 어떤 값을 받고 어떤 값을 반환하는지 정리합니다.

| 노드 | 입력 | 출력 |
| --- | --- | --- |
| 입력 검증 | user_id, message | is_valid, clean_message, reason |
| 문의 분류 | clean_message | category, priority |
| 검색어 변환 | clean_message, category | search_query |
| RAG 검색 | search_query | documents |
| 답변 생성 | clean_message, documents | draft_answer |
| 출력 검증 | draft_answer | final_answer, safety_status |

## 5. Loop, Fork-Join, 집계 필요 여부

n8n 또는 AIPP에서 반복/병렬 처리가 필요한 경우 아래를 작성합니다.

```text
반복 처리할 대상:
병렬로 실행할 작업:
결과를 합치는 기준:
실패한 항목 처리 방식:
```

예시:

```text
문의 5건을 한 번에 업로드하면 각 문의를 반복 처리한다.
문서 검색과 긴급도 판단은 병렬로 실행할 수 있다.
최종 결과는 ticket_id 기준으로 다시 합친다.
실패한 문의는 retry_count를 기록하고 담당자 검토로 보낸다.
```
