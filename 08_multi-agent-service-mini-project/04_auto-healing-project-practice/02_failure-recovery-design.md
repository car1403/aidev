# 02. Failure Recovery Design

장애 복구 설계는 “어떤 장애를 어떻게 감지하고 어떤 전략으로 복구할 것인가”를 정리하는 단계입니다.

## 장애 유형 예시

| 장애 유형 | 감지 기준 | 복구 전략 |
| --- | --- | --- |
| API 5xx | backend 응답 코드가 500번대 | retry, fallback 응답 |
| Timeout | 응답 시간이 기준보다 김 | retry, timeout 조정 |
| Rate Limit | 429 응답 | 대기 후 재시도, 캐시 사용 |
| LLM 응답 형식 오류 | JSON 파싱 실패, 필드 누락 | 재요청, structured output 강화 |
| Prompt Injection | 위험 지시문 포함 | 요청 차단, 감사 로그 기록 |
| Worker 실패 | 작업 중단, 로그 오류 | worker 재시작, 작업 재처리 |

## 복구 전략 예시

| 전략 | 의미 | 사용할 때 |
| --- | --- | --- |
| Retry | 같은 요청을 다시 시도 | 일시적인 네트워크 오류 |
| Restart | 서비스를 다시 시작 | worker나 컨테이너 상태 오류 |
| Reconnect | 연결을 다시 맺음 | 외부 API, DB, 큐 연결 끊김 |
| Fallback | 대체 응답이나 대체 경로 사용 | 원래 경로가 계속 실패 |
| Manual Review | 자동 복구 대신 확인 상태로 전환 | 위험하거나 반복 실패하는 경우 |

## 복구 설계 템플릿

```text
장애 이름:
발생 서비스:
감지 기준:
영향 범위:
1차 복구 전략:
2차 복구 전략:
복구 실패 시 처리:
검증 방법:
기록할 로그:
보안 주의사항:
```

## 중요한 기준

자동 복구는 항상 안전해야 합니다.

- 삭제, 권한 변경, 비용 발생 작업은 기본적으로 제한합니다.
- 민감 정보가 로그에 남지 않도록 합니다.
- 반복 실패 시 무한 재시도하지 않습니다.
- 복구가 성공했는지 Health Check로 검증합니다.
- 실패 결과도 monitor와 보고서에 남깁니다.
