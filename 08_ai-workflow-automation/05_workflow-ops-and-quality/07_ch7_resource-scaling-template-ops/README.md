# 07_ch7_resource-scaling-template-ops

AI 워크플로우의 리소스, 확장성, 재사용 가능한 템플릿 운영을 학습하는 챕터입니다.

이 챕터에서는 이미지 커리큘럼의 `Workflow 실행 흐름 최적화` 항목도 함께 다룹니다.
초보자는 최적화를 어렵게 느낄 수 있지만, 여기서는 "느린 단계 찾기", "불필요한 API 호출 줄이기", "반복해서 쓰는 흐름을 템플릿으로 만들기"처럼 눈에 보이는 기준부터 연습합니다.

## 학습 목표

- 워크플로우 실행에 필요한 CPU, GPU, 메모리, API 사용량을 추적할 수 있습니다.
- Workflow 실행 흐름 최적화가 필요한 구간을 찾을 수 있습니다.
- 수직 확장과 수평 확장의 차이를 이해합니다.
- 워크플로우 템플릿을 만들어 재사용성을 높이는 방법을 설계할 수 있습니다.
- 비용 최적화와 안정화 전략을 함께 고려할 수 있습니다.

## 기본 개념

| 항목 | 설명 |
| --- | --- |
| CPU | 일반 데이터 처리, 파싱, 변환 작업 |
| GPU | 로컬 모델 추론이나 대규모 처리 |
| Memory | 긴 Context, 대량 문서 처리 |
| API Usage | 외부 LLM/API 호출 횟수와 비용 |
| Vertical Scaling | 한 서비스의 자원을 키움 |
| Horizontal Scaling | 같은 서비스를 여러 개로 늘림 |
| Workflow Template | 반복 사용하는 워크플로우 구조를 템플릿화 |
| Workflow Flow Optimization | 오래 걸리거나 실패가 잦은 노드를 찾아 실행 순서, 조건, 호출 횟수를 개선 |

## Workflow 실행 흐름 최적화란?

Workflow 실행 흐름 최적화는 이미 만든 워크플로우를 더 빠르고 안정적으로 실행되도록 다듬는 과정입니다.

예를 들어 기술 지원 자동화 워크플로우가 다음 순서로 실행된다고 가정합니다.

```text
문의 입력
-> 모든 문서 검색
-> LLM 답변 생성
-> 긴급도 판단
-> 운영팀 알림
```

이 구조에서는 긴급하지 않은 문의에도 많은 문서를 검색하고 LLM을 호출할 수 있습니다.
최적화 후에는 다음처럼 바꿀 수 있습니다.

```text
문의 입력
-> 입력 검증
-> 긴급도와 문의 유형 먼저 판단
-> 필요한 경우에만 문서 검색
-> 필요한 경우에만 LLM 답변 생성
-> 긴급 건만 운영팀 알림
```

이렇게 바꾸면 실행 시간이 줄고, API 비용도 줄고, 오류가 발생할 지점도 줄어듭니다.

수업에서는 다음 기준으로 최적화 대상을 찾습니다.

```text
[ ] 실행 시간이 오래 걸리는 노드가 있는가?
[ ] 같은 API를 불필요하게 여러 번 호출하는가?
[ ] 실패해도 다음 단계로 넘어가면 안 되는 노드가 있는가?
[ ] 먼저 검증하면 비용을 줄일 수 있는 입력이 있는가?
[ ] 자주 반복되는 흐름을 템플릿으로 만들 수 있는가?
```
